# Architect+ v3 upgrade execution lessons

Use this reference when executing or repairing a SuperGoal that upgrades `chip-supergoal` itself, especially when the mission package path, skill self-tests, and generated launch markers can interfere with each other.

## Durable lessons

1. **Keep live mission packages outside the skill root during self-upgrades.**
   The skill test suite scans the skill tree for launch-marker invariants. A real generated `LAUNCH_GOAL.md` under `<installed-skill-dir>/.supergoal/...` can create false failures or marker-count drift. Prefer `<workspace-dir>/.supergoal/<mission>` for the active execution package while editing the installed skill root.

2. **Treat package-path drift as a controlled migration, not a blocker.**
   If the launch body references an unsafe/stale package path under the skill root, move or continue from a safe workspace package path, record the path change in `STATE.md`, and keep the skill implementation target as `<installed-skill-dir>`.

3. **Phase reports are the execution ledger for non-git skill roots.**
   When the skill root is not a git repository, replace commit/remote-head proof with per-phase reports containing changed-file hashes, command evidence, RPD review, and final audit JSON/MD. Do not invent commit proof.

4. **Security regression xfails must be temporary.**
   Initial Phase 0 tests may document known failures as expected-failure, but the fix phase must remove the expected-failure marker and prove the regression now passes normally.

5. **Private-boundary scans should ignore generated bytecode, not source.**
   Python tests can create `__pycache__` files containing string constants from security fixtures. Exclude `__pycache__` / `.pyc` from source privacy scans, while still scanning real source/templates/artifacts.

6. **Review integrity claims with adversarial repros, not only aggregate tests.**
   For compiler/validator hardening claims, make the independent review reproduce the specific promised guardrail. In alpha-style package reviews, add targeted probes for runtime-state preservation (`runtime/STATE.json`, `runtime/events.jsonl`, delivery/output dirs) and manifest integrity against real filesystem metadata (mode drift via `chmod`, generated-view drift, file-set drift). If the reviewer finds a gap, patch the candidate first, add regression tests, then apply to the installed skill.

7. **Final completion may be local-alpha when live GoalManager is unavailable.**
   A skipped `SUPERGOAL_HERMES_INTEGRATION=1` probe is a warning, not a blocker, if the hermetic GoalManager simulator, E2E, security, migration, deterministic build, and final audit all pass. Record the warning explicitly in final audit.

## Minimum final evidence shape

- `python3 -m unittest discover -s tests`
- `bash scripts/test.sh`
- `sgctl validate-contract ... --strict`
- compile twice and byte-compare immutable outputs
- `sgctl validate-package ... --strict`
- `sgctl migrate-v2 ...` plus validation of migrated contract
- `skill_view("chip-supergoal")` loadability proof
- `reports/final-audit.json` and `reports/final-audit.md`
- final markers in one response: `AUDIT_COMPLETE`, `SUPERGOAL_RUN_COMPLETE`, `Goal complete: yes`
