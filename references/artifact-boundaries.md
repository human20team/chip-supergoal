# SuperGoal Artifact Boundaries

This is the canonical contract for files generated by `chip-supergoal`. If another reference disagrees with this file, this file wins and the other reference is stale.

## Review pack v2

Chip-facing planning review uses **review_pack_v2**:

- `THINKING.md`
- `LOOP_DESIGN.md`
- `ROADMAP.md`
- `LAUNCH_GOAL.md`
- `RESEARCH.md` only when it exists and is non-empty

The review pack is sent during planner dispatch before `READY_TO_DISPATCH`. It is not a substitute for final deliverables and it is not an executor final-audit blocker after the user has already launched the goal.

## Boundary table

| Artifact | Owner | Stage | Visibility | Delivery default | Receipt / proof | Notes |
|---|---|---|---|---|---|---|
| `THINKING.md` | planner | planning | Chip review | yes | review receipt | Constraints, risks, assumptions, loaded context. |
| `RESEARCH.md` | planner | planning | Chip review when non-empty | conditional | review receipt | Omit when empty; never invent research. |
| `LOOP_DESIGN.md` | planner | loop-design gate | Chip review | yes | review receipt + validator | Execution harness: roles, gates, stop, budget, boundaries, recovery. |
| `ROADMAP.md` | planner | planning | Chip review | yes | review receipt | Phase map and acceptance contract. |
| `LAUNCH_GOAL.md` | planner | dispatch | Chip review / launch | yes | review receipt + single-marker scan | Only artifact with a real line starting `SUPERGOAL_GOAL_BODY:`. |
| `STATE.md` | planner then executor | execution | internal/status | no default review delivery | state readback | Executor-owned after launch; terminal state must be `DONE`/`COMPLETE` before final markers. |
| `PROTOCOL.md` | planner | execution | internal executor contract | no default review delivery | copied file on disk | Must be self-contained inside `.supergoal/`; no external `references/` dependency. |
| `phases/phase-N.md` | planner then executor | execution | internal executor contract | no default review delivery | phase validation | Strict schema; each phase validated before dispatch. |
| `scripts/repo-state.sh` | planner | execution helper | internal | no default review delivery | shell syntax + functional tests | Copied into `.supergoal/scripts/`. |
| `reports/final-audit.md` | executor | final audit | final/status | only if final artifacts requested | final audit markers | Must include `AUDIT_COMPLETE` and `SUPERGOAL_RUN_COMPLETE` only after verification. |
| final artifact bundle | executor | delivery | Chip delivery when requested | conditional | final receipt | Separate from planning review pack. |

## Receipts

Review delivery receipts prove planner dispatch, not product completion. They should include:

- `ok: true`
- `sent: true`
- `pack_version: "review_pack_v2"`
- target chat/thread metadata when available
- sent file list
- per-file `sha256` and byte size when available

Final delivery receipts prove executor-side final artifact delivery. They should not be conflated with review receipts.

## Hard invariants

- `LAUNCH_GOAL.md` is the only launch surface with a real `SUPERGOAL_GOAL_BODY:` line.
- `LOOP_DESIGN.md` is mandatory for every non-trivial package.
- `SUPERGOAL_TURN_YIELD` is a forced-yield/blocker marker, not a courtesy phase boundary.
- Planning delivery failure blocks `READY_TO_DISPATCH`; it should not retroactively block final audit if the user explicitly launched the goal anyway.
- Historical references may mention older three-file delivery behavior only when marked as archived/legacy, not as active policy.
